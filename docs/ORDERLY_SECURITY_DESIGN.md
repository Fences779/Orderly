# Orderly 本地安全设计（P0 安全基线）

本文档记录 Orderly 桌面端的本地数据安全设计，作为发布前安全复核与审计的可追溯依据。它描述当前 Commerce（通用经营）系统所沿用的 P0 安全行为；任何改动若与本文所述安全行为冲突，一律以安全行为为准。

## 1. 数据落盘加密（SQLCipher 全库加密）

- 每个账号的业务数据库与启动器数据库均使用 SQLCipher 进行整库加密。
- 连接通过带密钥提供者的连接工厂打开：密钥在打开连接后立即下发（`PRAGMA key`），用完即清零；不带密钥的连接工厂仅用于测试或显式无加密场景。
- 加密密钥来源于当前会话的数据密钥（`LocalSessionContext.DataKey`，32 字节）。该密钥不落盘为明文。
- 启动器数据库密钥由操作系统级保护（DPAPI，CurrentUser 作用域）保护后存储。
- 既有未加密数据库在打开前由加密迁移器原地升级为加密格式，过程非破坏性。

## 2. 字段级敏感数据加密

- 在整库加密之外，敏感字段（如客户名称、联系方式、电话、备注、金额、时间等）额外做字段级 AES-GCM 加密，密文写入对应的 `*Ciphertext` 列，明文列在写入路径被清空。
- 密文带版本前缀（`v1:`）。每个字段使用绑定的“关联数据”参与认证，关联数据格式为 `表名.密文列|row:行号`，确保密文不可跨行/跨字段重放。
- 字段加密 fail-closed：若写入路径检测到空操作加密器，绝不回写明文（以脱敏占位符替代），避免敏感字段明文落盘。
- 升级旧库时由后台回填将明文迁移为密文并清空明文列；该回填对未加密旧库与已加密库均收敛。

## 3. 本地账号与启动入口安全

- 多账号结构、启动器数据库与身份目录统一位于 `%LocalAppData%\Orderly` 之下。
- 账号数据库路径安全校验（账号库必须位于规范的 `accounts/{accountId}/orderly.db`）在登录路径强制执行，禁止指向任意位置的数据库文件。
- 关键文件与目录拒绝符号链接 / 重解析点，防止通过链接越权读写。
- QA / Demo 等特权启动入口仅在 `ORDERLY_RUNTIME_ENV`（或 `DOTNET_ENVIRONMENT`）为 development/dev/qa/test/local 时允许，且必须显式设置 `ORDERLY_ENABLE_PRIVILEGED_QA_STARTUP=1`；QA 数据库路径必须显式经 `ORDERLY_QA_DB_PATH` 指定，并位于 `ORDERLY_QA_DATA_ROOT` 指定的隔离目录之下，绝不回落到真实默认数据库。

## 4. 备份与恢复安全

- 备份导出为密文落盘：清单经校验和与完整性标签处理后再整体加密写盘；磁盘上的备份文件不以明文暴露清单内容（计数、校验和等）。
- 备份元数据仅保存文件名而非绝对路径，避免泄露本地路径。
- 恢复前提供预览与校验：校验失败或被篡改的备份会被拒绝；恢复目标状态（空库 / QA 库 / 非空生产库）参与放行判定。
- 已知风险：启动器数据库与业务数据库为两库分步提交，跨两库的恢复不是单一原子事务。该风险已登记，恢复任一库失败时应人工核对两库一致性。

## 5. 旧数据迁移安全（Commerce）

- 旧 CRM 数据到 Commerce 模型的迁移是 backup-first、非破坏性、幂等的：迁移前先完整备份源库；只读旧表、只写 Commerce 表；以确定性目标标识做 upsert，重复执行不产生重复。
- 备份失败即中止迁移并保留旧数据；迁移按完成标记保证仅执行一次，失败 / 中断不写完成标记，下次启动重试。
- 迁移读取旧表敏感字段时优先解密 `*Ciphertext` 列（与字段加密一致的关联数据方案），密文为空时回落到明文列，使升级前后两种磁盘状态都能正确迁移真实数据。

## 6. 业务写入一致性

- 核心业务写入（如订单完成、收款）在单一 `Core_Write_Transaction` 内完成：收款记录与其现金流流水同事务写入，任一步失败整体回滚，杜绝孤立现金流。
- 幂等业务键（收款、现金流、库存变动、经营洞察、指标快照）在存储层以 `(WorkspaceId, BusinessKey)` 分区唯一索引约束；服务层遇唯一冲突转化为幂等成功，并发下不产生重复。

## 7. 发布前安全门禁

发布前重新运行 P0 安全自动化套件，要求相对基线零失败、零新增跳过。详见 `docs/ORDERLY_RELEASE_CHECKLIST.md` 与 `docs/ORDERLY_QA_AUTOMATION.md`。
