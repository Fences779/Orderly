# Orderly Cloud Sync v1：需求与架构定义

## 1. 项目定义

Orderly Cloud Sync 是 Orderly 多客户端之间的中心化数据同步系统。

```text
Orderly A / B / C
    ⇅
Orderly Cloud Sync Server
    ⇅
Orderly A / B / C
```

系统目标不是单纯上传和下载文件，而是让多个 Orderly 客户端围绕同一套云端主数据持续同步：

* 客户端可向云端提交新增、修改、归档等业务变更。
* 云端保存全局主数据、版本、审计记录和附件。
* 在线客户端在数秒内收到其他客户端的变更。
* 离线客户端恢复网络后自动补齐全部缺失变更。
* 云端是最终可信状态源；客户端本地库是可离线工作的副本。

---

## 2. v1 总体边界

### 2.1 数据拓扑

云端采用统一的全局主数据池。

所有 Orderly 用户理论上连接同一套云端系统，但每个客户端实际可访问哪些数据、能执行哪些操作，取决于云端下发的权限与分发范围。

```text
云端：完整主数据池
    ↓ 按权限范围分发
客户端：仅获得被授权的数据与操作能力
```

`Workspace / Scope` 的作用是权限与数据分发边界，不代表每个空间都必须是一套独立数据库。

### 2.2 安全原则

* 客户端本地配置可以进一步收紧展示或操作权限。
* 客户端本地配置不得扩大云端授权。
* 所有读取、写入、附件下载、历史恢复、设备授权，均由云端二次校验。
* 任何绕过本地界面直接调用 API 的行为，也必须受云端权限控制。

---

## 3. 角色模型

系统存在三类不同性质的管理员，必须严格区分。

### 3.1 云端管理员

云端管理员仅包括：

* 创建者
* 投资方

职责：

* 审核新用户申请。
* 配置用户可访问的数据范围与权限。
* 审核新增设备及多设备使用资格。
* 管理全局成员、空间、权限、配额、保留策略。
* 查看数据内容以排查问题。
* 执行数据恢复、人工修复、过期数据最终删除确认。
* 查看全部审计记录。

所有云端管理员操作必须留痕，包括：

```text
操作者
操作时间
操作对象
修改前数据
修改后数据
操作原因
来源 IP / 设备
结果
```

### 3.2 空间角色

每个 Workspace / Scope 内支持以下角色：

| 角色     | 权限                  |
| ------ | ------------------- |
| Owner  | 空间最高业务权限，可管理成员和业务配置 |
| Admin  | 管理成员、部分配置和业务数据      |
| Editor | 创建、编辑、归档被授权业务数据     |
| Viewer | 只读访问被授权数据           |

空间角色不等于云端管理员。

普通空间 Owner 或 Admin 不自动拥有全局云端管理权。

### 3.3 本地管理员

本地管理员是某一台 Windows 设备、某一个 Windows 账户下的 Orderly 本地管理员。

特点：

* 一台电脑可存在多个 Windows 账户。
* 每个 Windows 账户内的 Orderly 可有独立本地管理员。
* 本地管理员负责本机 Orderly 的本地配置、可见模块、本机操作策略等。
* 本地管理员不能自行扩大云端权限、批准设备、修改云端成员权限或绕过服务端校验。

本地管理员是“本机环境管理员”，不是“云端系统管理员”。

---

## 4. 用户、申请与设备管理

### 4.1 新用户进入流程

v1 不开放公开注册。

标准流程：

```text
邀请码 / 申请入口
→ 用户提交申请
→ 云端管理员核实
→ 云端管理员人工配置空间、范围、角色、权限
→ 审批通过
→ 用户完成用户名与密码创建
→ 首台设备注册并激活
```

审批结果、权限配置和审批人必须记录审计日志。

### 4.2 登录方式

v1 登录方式：

```text
邀请码 + 用户名 + 密码
```

账号创建后，账号状态至少包括：

```text
Pending
Active
Rejected
Suspended
Disabled
```

### 4.3 设备模型

每台客户端设备需要独立设备身份：

```text
DeviceId：系统按固定规则生成，不允许重复
DeviceName：用户可自行命名
DeviceStatus：Pending / Approved / Revoked / Disabled
```

设备授权规则：

* 默认只允许用户使用已批准的设备。
* 新设备登录或申请多设备使用时，必须由云端管理员批准。
* 云端管理员批准、拒绝、撤销设备均必须留痕。
* 用户可在账户页查看并撤销自己的其他设备。
* 云端管理员可强制撤销任意设备。

---

## 5. 数据同步范围

v1 同步范围包括：

* 全部核心业务数据。
* 可跨设备使用的配置。
* 业务附件、图片、文件。
* 业务审计记录。
* 必要的同步元数据。
* 必要的设备状态与版本信息。

### 5.1 本地偏好

个人偏好采用“选择性同步”：

| 类别           | 规则        |
| ------------ | --------- |
| 主题、语言、个人快捷键  | 可同步       |
| 窗口位置、屏幕布局    | 默认本地保存    |
| 临时缓存、索引、下载缓存 | 不同步       |
| 排序偏好、展示偏好    | 视业务决定是否同步 |

### 5.2 附件规则

附件支持同步，但必须独立管理：

* 附件不直接存进 PostgreSQL 数据库字段。
* 附件存入对象存储。
* PostgreSQL 仅保存附件元数据、权限、哈希、版本、归属和审计信息。
* 每个 Workspace / Scope 配置附件容量配额。
* 上传、下载、替换、归档、过期删除均需审计。
* 附件读取必须走云端授权或短时签名地址，不允许公开直链。

### 5.3 日志规则

日志分为两类：

| 类型      | 处理方式                         |
| ------- | ---------------------------- |
| 业务审计日志  | 云端长期保存，按权限查看                 |
| 客户端诊断日志 | 默认不向普通用户同步；发生故障时按需上传给云端管理员排查 |

## 6. 数据生命周期与删除策略

v1 原则：**普通用户不可永久删除数据。**

### 6.1 常规删除

用户执行“删除”时，实际行为是：

```text
Archive / Soft Delete
```

数据从普通视图隐藏，但云端仍保留：

* 原始内容。
* 归档时间。
* 归档人。
* 归档原因。
* 历史版本。
* 恢复记录。

### 6.2 永久删除

永久删除必须同时满足：

```text
达到设定保留期限
+
云端管理员确认
+
全过程审计留痕
```

永久删除不允许静默执行。

### 6.3 历史版本

核心业务实体保留一段时间内的历史变更，可用于：

* 数据恢复。
* 冲突排查。
* 审计追踪。
* 管理员修复。
* 用户误操作恢复。

---

## 7. 同步模型

### 7.1 核心链路

```text
本地修改
→ 写入本地数据库
→ 生成待上传变更 Outbox
→ 上传至云端
→ 云端校验权限、版本和幂等性
→ 云端写入主数据与变更日志
→ 云端通知在线客户端
→ 在线客户端拉取增量变更
→ 离线客户端重连后按 Cursor 补齐
```

### 7.2 实时体验目标

v1 实时要求：

* 在线客户端：其他客户端提交变更后，数秒内自动更新。
* 离线客户端：恢复网络后自动补齐。
* 客户端不依赖“必须收到实时消息”才能保持正确。
* 实时通道只负责提醒“有新变更”；实际数据仍必须通过增量拉取确认。

推荐实现：

```text
ASP.NET Core SignalR：实时通知
HTTPS Sync API：上传、拉取、补齐
PostgreSQL：主数据、版本、变更流
```

### 7.3 客户端同步模块

Orderly 客户端必须有独立同步模块，不允许同步逻辑散落在页面代码中。

```text
Orderly.Sync
├─ Local Store Adapter
├─ Outbox
├─ Sync Cursor
├─ Pull Processor
├─ Conflict Handler
├─ Realtime Listener
├─ Retry / Backoff
└─ Permission Distribution Cache
```

---

## 8. 冲突与一致性规则

### 8.1 云端优先

云端为唯一可信状态源。

客户端不可使用旧版本直接覆盖云端新版本。

### 8.2 实体版本控制

每条可同步实体至少包含：

```text
EntityId
WorkspaceId / ScopeId
Version
CreatedAt
CreatedBy
UpdatedAt
UpdatedBy
DeletedAt / ArchivedAt
```

客户端提交修改时必须携带：

```text
BaseVersion
ChangedFields
IdempotencyKey
```

### 8.3 同一字段冲突

两端同时修改同一字段时：

```text
先成功提交的一方生效
后提交的一方被服务端拒绝
客户端拉取最新版本
用户看到冲突提示并决定处理方式
```

不得静默覆盖。

### 8.4 不同字段并发修改

例如：

```text
A 修改标题
B 修改截止日期
```

系统允许服务端进行字段级自动合并。

前提：

* 两次修改基于同一旧版本。
* 修改字段不重叠。
* 实体结构允许字段级补丁更新。

合并成功后生成新版本并通知所有客户端。

### 8.5 重复请求

所有写入请求必须带 `IdempotencyKey`。

同一请求因网络问题重复发送时：

* 服务端只能执行一次实际写入。
* 后续重复请求返回原处理结果。
* 不允许重复创建实体、重复上传变更或重复写审计记录。

---

## 9. 权限分发与访问控制

### 9.1 权限分发

云端管理员为用户配置：

```text
可访问 Scope / Workspace
空间角色
可读数据范围
可写数据范围
模块可见性
附件配额
设备授权状态
特殊限制
```

客户端获得权限快照后，可据此限制本地 UI、菜单、模块和操作入口。

### 9.2 服务端强制校验

每一次请求均由服务端校验：

```text
用户身份
设备状态
令牌有效性
空间归属
实体归属
角色权限
字段级权限
附件权限
当前版本
```

客户端隐藏按钮不等于有安全性；服务端拒绝才是最终限制。

---

## 10. 管理后台

v1 必须包含云端管理后台。

管理后台至少支持：

* 用户申请审核。
* 用户启用、禁用、冻结。
* 邀请码管理。
* Workspace / Scope 管理。
* 成员、角色、权限范围配置。
* 设备审批、撤销、禁用。
* 附件配额配置。
* 数据查看与故障排查。
* 历史版本查看与恢复。
* 归档数据查看与永久删除确认。
* 审计日志查询与筛选。
* 服务健康状态、备份状态、同步异常查看。

云端管理员查看业务正文、附件或用户数据时，也必须记录访问审计。

## 11. 基础设施目标架构

v1 推荐基础架构：

```text
域名
  ↓
HTTPS Reverse Proxy（Caddy / Nginx）
  ↓
ASP.NET Core API + SignalR
  ↓
PostgreSQL
  ↓
Object Storage（附件）
  ↓
异地备份
```

部署建议：

```text
Docker Compose
Linux VPS
PostgreSQL
ASP.NET Core
SignalR
Caddy / Nginx
对象存储
自动备份任务
健康检查
日志收集
```

初始服务器建议按以下级别准备：

```text
2 vCPU
4 GB RAM
80 GB SSD 起步
```

原因：

* 除 API 外还需运行 PostgreSQL。
* 需要承载实时连接、管理后台、备份任务和附件元数据。
* 2 GB 内存在后续加对象存储、监控和后台任务时余量偏小。

---

## 12. 备份、恢复与可用性

### 12.1 备份标准

必须具备：

```text
每日自动数据库备份
保留至少 30 天
附件元数据与附件文件分别备份
定期验证可恢复性
异地备份副本
```

备份完成不等于有效，必须定期实际恢复到测试环境验证。

### 12.2 宕机预期

v1 不要求双机高可用，但要求：

* 容器或服务异常后自动重启。
* 服务器重启后服务自动恢复。
* 客户端具备离线队列和自动重试。
* 短暂宕机期间不丢失客户端本地未上传数据。
* 服务恢复后客户端自动重连、补齐变更。

---

## 13. 审计要求

以下行为必须进入不可随意删除的审计链：

```text
登录成功 / 失败
用户申请与审批
设备申请、批准、拒绝、撤销
权限、角色、Scope 修改
业务创建、修改、归档、恢复
永久删除确认
附件上传、下载、替换、归档
管理员查看敏感业务内容
管理员直接修复数据
备份失败、恢复操作
同步冲突与服务端拒绝写入
```

审计至少包含：

```text
Actor
ActorRole
DeviceId
TargetType
TargetId
Action
Before
After
Reason
Timestamp
IP
Result
CorrelationId
```

## 14. v1 验收标准

v1 只有在以下全部通过后才允许给内部 5 人使用。

### 14.1 用户与权限

* 用户可通过邀请码发起申请。
* 云端管理员可审核、分配空间和权限。
* 权限配置完成后，客户端只获得授权范围内的数据。
* 修改本地客户端配置无法扩大云端数据访问范围。
* 云端管理员审批和权限变更均可审计追溯。

### 14.2 设备与账号

* 首台设备可激活。
* 后续设备需云端管理员批准。
* 用户可查看、命名和移除已登录设备。
* 云端管理员可撤销设备，撤销后该设备立即失去云端访问资格。

### 14.3 同步闭环

* A 创建数据后，B 在线时数秒内自动看到更新。
* B 离线期间，A 连续修改数据；B 重连后可自动补齐。
* A/B 同时修改同一字段时，后提交方会收到明确冲突结果。
* A/B 修改不同字段时，系统可自动字段级合并。
* 网络抖动导致请求重发时，不会产生重复数据。
* 客户端异常退出后，未上传变更仍保留在 Outbox，重启后可继续同步。

### 14.4 数据与附件

* 所有业务数据可按版本追溯。
* 普通用户不能永久删除数据。
* 归档数据可恢复。
* 永久删除必须经过保留期与云端管理员确认。
* 附件按 Workspace 配额受控。
* 未授权用户无法下载附件。

### 14.5 运维与恢复

* 数据库每日备份。
* 备份保留 30 天。
* 至少完成一次真实恢复演练。
* 服务重启后 API、SignalR、数据库自动恢复。
* 管理后台可查看基本健康状态、同步异常和审计日志。

## 15. 施工策略

对外发布策略：

```text
不做半成品公开上线。
全部同步能力、管理后台、权限、审计、备份和客户端闭环完成后，再向内部 5 人开放。
```

工程施工仍分阶段推进：

```text
P0：需求与协议冻结
P1：数据库、权限模型、同步契约
P2：服务器、域名、HTTPS、数据库、备份
P3：云端 API、SignalR、对象存储、审计
P4：Orderly 客户端同步引擎
P5：云端管理后台
P6：A/B/C 多端联调与异常测试
P7：内部 5 人一次性上线
```

## 16. 当前正式结论

Orderly Cloud Sync v1 的定位是：

```text
一个中心化、版本化、可审计、可离线补齐、
支持实时分发和多设备审批的 Orderly 云端同步系统。
```

它不是单纯的“上传下载服务”，也不是客户端之间的点对点同步。

云端负责：

```text
身份
设备
权限
数据主状态
版本
冲突判定
变更分发
附件
审计
备份
恢复
```

客户端负责：

```text
本地使用体验
本地数据库
离线变更队列
实时通知接收
增量拉取
冲突展示
本地权限界面限制
```
