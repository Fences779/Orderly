# Orderly 团队云端正式版人工操作手册（给你）

这份文档只写你需要人工做的事：买什么、配什么、怎么上线、怎么日常管。代码施工看 `docs/ORDERLY_TEAM_CLOUD_AI_IMPLEMENTATION_PLAN.md`。

## 1. 最终你会得到什么

正式版 Orderly 会变成：

```text
你电脑上的 Orderly
员工电脑上的 Orderly
投资方电脑上的 Orderly
        ↓
阿里云 Orderly 中心服务端
        ↓
PostgreSQL 中心经营数据库
```

结果：

- 三个人看同一份经营数据。
- 修改后实时同步。
- 投资方和你同样最高权限。
- 员工受限，只能做该做的事。
- 员工看不到成本、利润、毛利率。
- 员工不能导出。
- 员工不能直接改价，只能提交申请。
- 归档不是删除，只是不在默认列表显示。
- 操作都有记录。
- 数据每天备份，保留 30 天。

## 2. 云资源购买建议

第一版数据少，人也少，不用买贵的。

建议买阿里云：

```text
地域：华东 1（杭州）
服务器：轻量应用服务器或 ECS
配置：2 核 4G
系统盘：60G 起
系统：Ubuntu 22.04 LTS 或 Ubuntu 24.04 LTS
带宽：3M 起
数据库：同机 PostgreSQL
```

预算：

```text
服务器：约 300 - 800 元 / 年
域名：约 50 - 100 元 / 年
SSL：0 元
备份存储：第一版本机 + 阿里云 OSS 异地备份，少量数据通常很低
```

不建议第一版买独立云数据库。你现在数据少，同机 PostgreSQL 足够，先把系统跑通更重要。

上线前你要准备：

```text
域名：例如 orderly.xxx.com
阿里云账号
服务器公网 IP
服务器 root 或 sudo 账号
首个管理员用户名
首个管理员显示名
首个管理员初始密码
投资方用户名
投资方显示名
员工用户名
员工显示名
```

建议账号标签：

```text
你：运营负责人 / Admin
投资方：投资方 / Admin
员工：员工 / Employee
```

## 4. 安全组和网络

阿里云安全组只开放：

```text
22    SSH，仅你自己的 IP 或临时开放
80    HTTP，用于申请 HTTPS 证书
443   HTTPS，Orderly 正式访问入口
```

不要开放：

```text
5432  PostgreSQL
5000  ASP.NET 内部端口
```

数据库只能让服务器本机访问，不给公网连。

## 5. 域名和 HTTPS

域名 DNS：

```text
orderly.xxx.com -> 服务器公网 IP
```

HTTPS 建议用 Caddy 自动申请证书。

你只需要保证：

- 域名解析到服务器。
- 80 和 443 端口开放。
- 服务端部署文件里写对域名。

## 6. 正式部署建议目录

服务器上统一放这里：

```text
/opt/orderly
/opt/orderly/app
/opt/orderly/data/postgres
/opt/orderly/backups
/opt/orderly/exports
/opt/orderly/logs
```

备份目录：

```text
/opt/orderly/backups
```

保留 30 天。

## 7. 环境变量

上线时需要配置：

```text
ORDERLY_PUBLIC_URL=https://orderly.xxx.com
ORDERLY_POSTGRES_HOST=postgres
ORDERLY_POSTGRES_PORT=5432
ORDERLY_POSTGRES_DB=orderly
ORDERLY_POSTGRES_USER=orderly
ORDERLY_POSTGRES_PASSWORD=一段强密码
ORDERLY_JWT_SIGNING_KEY=至少32位随机字符串
ORDERLY_BACKUP_RETENTION_DAYS=30
ORDERLY_BOOTSTRAP_ADMIN_TOKEN=一次性初始化密钥
ORDERLY_ALLOWED_ORIGINS=https://orderly.xxx.com
```

密码要求：

- 不要用生日、手机号、公司名。
- 不要发给别人。
- 不要截图发群。
- 保存到你的密码管理器或本地加密笔记。

## 8. 第一次上线步骤

代码施工完成后，人工上线大概按这个顺序：

1. 买阿里云服务器。
2. 配安全组。
3. 配域名 DNS。
4. SSH 登录服务器。
5. 安装 Docker 和 Docker Compose。
6. 上传或拉取 Orderly 部署包。
7. 填 `.env`。
8. 启动服务。
9. 打开 `/health` 检查服务。
10. 初始化首个管理员账号。
11. 创建投资方管理员账号。
12. 创建员工账号。
13. 从你的电脑上传当前模板数据做测试。
14. 三台电脑分别登录测试。
15. 测试实时同步。
16. 测试备份文件是否生成。
17. 测试导出 ZIP。

## 9. 首批账号怎么建

账号建议：

```text
你：
  角色：Admin
  标签：运营负责人
  权限：最高权限

投资方：
  角色：Admin
  标签：投资方
  权限：最高权限

员工：
  角色：Employee
  标签：员工
  权限：受限
```

员工账号不是“本机登录账号”那么简单，它代表公司里的一个操作人员。

系统必须记录：

- 谁创建的账号。
- 什么时候创建。
- 给了什么角色和业务标签。
- 谁改过角色、标签或启停状态。
- 是否停用。

## 10. 员工能做什么

员工可做：

- 看订单。
- 新建订单。
- 处理订单。
- 完成订单。
- 确认收款状态。
- 确认履约状态。
- 看商品售价。
- 提交商品改价申请。
- 入库。
- 出库。
- 盘点调整。
- 看客户资料。
- 写客户备注。
- 看订单相关收款状态。
- 归档自己创建或分配给自己的订单、客户、待办。

员工不能做：

- 看成本。
- 看利润。
- 看毛利率。
- 看经营建议。
- 新增现金流。
- 修改现金流。
- 导出数据。
- 直接改商品价格。
- 真删除数据。
- 改系统设置。

## 11. 改价流程

员工不能直接改价。

正常流程：

```text
员工提交改价申请
    ↓
你或投资方看申请
    ↓
根据市场价决定通过或驳回
    ↓
通过后商品价格生效
    ↓
系统记录全过程
```

记录内容：

- 申请人。
- 申请时间。
- 商品。
- 原价格。
- 建议新价格。
- 申请理由。
- 审核人。
- 审核时间。
- 审核结果。

## 12. 库存调整规则

员工可以做库存操作，但每次都留痕。

入库、出库、盘点调整都要记录：

- 调整前数量。
- 调整后数量。
- 差异数量。
- 操作人。
- 操作时间。
- 原因。
- 备注。
- 是否盘点。

盘点调整不要当成普通“改数量”，必须走盘点记录。

## 13. 归档规则

归档不是删除。

归档后的数据：

- 默认列表不显示。
- 数据还在。
- 操作日志还在。
- 管理员可以查看。
- 管理员可以恢复。

员工：

- 可以归档自己创建或分配给自己的订单、客户、待办。
- 归档要写原因。

你和投资方：

- 可以查看全部归档。
- 可以恢复归档。
- 可以追溯谁归档的。

## 14. 冲突规则

多人同时改同一条数据时：

```text
第一个提交成功的人生效。
第二个人提交时失败。
系统提示数据已经变化。
第二个人刷新后重新确认。
```

不会静默覆盖。

打开数据时会提示：

- 谁正在查看。
- 谁正在编辑。
- 最近谁改过。
- 什么时间改的。

## 15. 断网规则

断网后：

- 可以看最近缓存的数据。
- 不能正式修改数据。
- 可以做应急预处理草稿。

允许的应急草稿：

- 订单备注。
- 客户备注。
- 待办状态。
- 订单状态。

不允许离线正式提交：

- 库存调整。
- 现金流。
- 改价申请。
- 导出。

联网后：

- 系统列出草稿。
- 你逐条确认是否提交。
- 如果云端数据已变化，系统提示冲突。

## 16. Excel 导出

导出只给你和投资方。

员工不能导出。

导出结果是一个 ZIP，里面按类拆成多份：

```text
orders.xlsx
products.xlsx
inventory.xlsx
customers.xlsx
cashflow.xlsx
price-change-requests.xlsx
audit-logs.xlsx
archive.xlsx
```

不要做成一个大杂表，不好查，也不好给别人看。

## 17. 备份和恢复

备份规则：

```text
每天自动备份一次
每天上传阿里云 OSS 一份
保留 30 天
超过 30 天自动清理
```

你要定期检查：

- 备份目录有没有新文件。
- OSS 有没有同步的新文件。
- 文件大小是不是明显异常。
- 每周自动恢复检查是否成功。
- 一个月至少做一次恢复演练。

第一版就要做：

- 本机 `pg_dump`。
- OSS 异地备份。
- 每周自动恢复检查。
- 每月人工恢复演练。

## 18. 日常管理

每天看：

- 服务是否正常。
- 昨天备份是否生成。
- 是否有失败的导出。
- 是否有异常登录。
- 是否有员工高频归档或库存调整。

每周看：

- 操作日志。
- 库存调整记录。
- 改价申请记录。
- 备份是否连续。

有人离职：

1. 停用账号。
2. 不删除账号。
3. 保留历史操作。
4. 检查最近导出记录。
5. 检查最近归档和库存调整。

## 19. 投资方移动办公

投资方经常国内外移动，所以要注意：

- 必须走 HTTPS。
- 不要裸 IP 登录，使用域名。
- 账号密码不要发聊天软件。
- 如果以后国外访问慢，再考虑加海外访问优化。

第一版不做手机端，电脑端远程登录即可。

## 20. 上线验收清单

上线时你检查这些：

- 你能登录。
- 投资方能登录。
- 员工能登录。
- 你和投资方看到的数据一致。
- 员工看不到成本、利润、毛利率。
- 员工看不到经营建议。
- 员工不能导出。
- 员工提交改价申请后，你能审批。
- 审批通过后，员工电脑商品价格刷新。
- 员工盘点调整后，有调整前后记录。
- 一个人编辑订单时，另一个人看到提示。
- 两个人同时改同一条订单，第二个人会冲突失败。
- 断网后能看缓存，不能正式提交。
- 联网后能提交应急草稿。
- 管理员能导出 ZIP。
- 备份文件生成。
- 操作日志能看到关键动作。

## 21. 正式开工前补充操作规则

这些是正式使用时防事故的硬规则，不能省。

### 21.1 本地旧数据导入

本地 SQLite 数据不能手工复制。

正确流程：

1. 你用管理员账号登录当前 Orderly。
2. 点击“一次性导入云端”。
3. 系统先做预检查。
4. 系统生成导入报告。
5. 你确认报告。
6. 系统才开始导入云端 PostgreSQL。
7. 成功后写导入审计。

导入身份规则：

- 每个本地数据库有一个永久 `SourceInstanceId`，首次创建后固定不变。
- `SourceFingerprint` 只用来确认 DryRun 到 Commit 之间本地数据没变。
- Commit 前会重新核对 `SourceFingerprint`，如果变了，拒绝导入，必须重新 DryRun。
- 当前登录管理员所在的云端工作区就是导入目标，客户端不能自己指定别的工作区。
- 一个本地工作区第一版只导入到一个云端工作区，不自动拆分多个云端工作区。

如果导入失败：

- 整批回滚。
- 不产生半批数据。
- 不重复生成订单、库存、客户。
- 操作日志记录失败原因。

### 21.2 实时同步不是只靠 SignalR

SignalR 只负责提醒。

真正同步逻辑是：

```text
服务端给每次变化一个连续编号
客户端保存最后同步到的编号
断网或休眠后重连
客户端按编号补拉漏掉的数据
```

这里的连续编号不是 PostgreSQL 全局自增号，而是每个工作区单独维护的已提交同步序号。服务端每次写数据时会锁住工作区同步状态，递增序号，再和业务数据一起提交。

所以即使电脑休眠、网络卡顿、SignalR 消息漏了，也能通过补拉恢复。

FullResync 时，本地不会直接覆盖旧缓存。

正确流程：

1. snapshot 先写入 staging cache。
2. 全部分页拉完并校验通过。
3. 原子替换该工作区正式缓存。
4. 再从 `SnapshotSequence` 之后补拉 changes。
5. 已归档、已删除或已经失去权限的数据，不允许继续残留在客户端正式缓存里。

### 21.3 防重复点击

所有关键写操作都有 `ClientRequestId` 防重复。

这能避免：

- 连点完成订单导致重复完成。
- 网络卡顿导致重复扣库存。
- 重复生成现金流。
- 重复提交改价申请。

同一个人、同一个工作区、同一个动作、同一个请求 ID，只执行一次。

如果第一次请求正在处理，重复请求会等数据库事务结束；首次请求提交成功后返回第一次结果，首次请求回滚后后续请求可以安全重试。

异步导出也是一样：只在事务内创建导出任务，重复请求返回同一个 `ExportJobId`，不把导出处理进度混进幂等规则里。

### 21.4 库存不会扣成负数

库存扣减必须在 PostgreSQL 事务里硬保证。

完成订单时必须同时完成：

- 检查库存。
- 扣库存。
- 写库存流水。
- 写库存审计。
- 完成订单。

库存不足时，整笔订单失败，不会部分成功。

同一订单涉及多个库存项时，系统会按固定顺序锁库存，遇到数据库死锁或序列化失败时只做有限次数安全重试。

### 21.5 账号和登录

正式版必须有：

- AccessToken。
- RefreshToken。
- RefreshToken 刷新接口。
- 改密码后旧登录全部失效。
- 停用员工后该员工所有登录立刻失效。
- 密码重置流程。
- token 不明文保存在电脑上，使用 Windows 安全存储。

安全限制：

- 不能停用最后一个管理员。
- 管理员不能把自己锁死。
- 每次 API 请求和实时连接都会检查账号是否仍启用、token 版本是否仍有效。
- 旧 refresh token 被重复使用会触发重放保护，并按 `TokenFamilyId` 撤销整条会话链，要求重新登录。
- 登录失败会限流或锁定。
- 密码哈希使用 Argon2id 等明确安全算法，不能自定义弱哈希。
- 密码、token、密钥不能写进操作日志。

### 21.6 权限第一版固定

第一版只做固定角色：

```text
Admin
Employee
```

不做复杂自定义权限。

员工“负责的数据”按下面判断：

- 数据由员工创建。
- 或数据分配给该员工。

第一版只有订单、客户、待办支持员工按“负责”归档。

员工不能归档商品、库存项、库存流水、现金流、经营建议和系统配置。

### 21.7 金额、数量和历史价格

数据库里金额和库存数量不能用文本。

规则：

- 金额保留 2 位。
- 数量保留 4 位。
- 服务端统一存 UTC。
- 客户端显示北京时间。
- 商品改价只影响未来订单。
- 历史订单保留当时成交价，不随商品价格变化。
- `OrderItem.UnitPrice`、`OrderItem.UnitCost`、`OrderItem.LineTotal` 在创建订单项时写入并固定，除非走明确、受审计的订单项修改操作。

### 21.8 数据库升级和回滚

数据库升级不能只靠自动建表。

正式流程：

1. 升级前自动备份。
2. 备份上传 OSS。
3. 执行数据库迁移。
4. 迁移成功再启动新版本。
5. 迁移失败就停住，不进入可写状态。

回滚规则：

- 第一版禁止破坏性数据库迁移。
- 如果应用要回滚，数据库必须兼容旧版本。
- 如果数据库已经不可逆变化，只能从备份恢复。

注意：备份不能只放服务器本机。第一版就要每天上传 OSS，每周自动恢复检查，每月人工恢复演练。

### 21.9 导出和列表

导出规则：

- 导出文件 24 小时后自动删除。
- 下载导出文件必须重新鉴权。
- 不生成公开下载链接。
- 导出失败最多重试 2 次。
- 导出目录有磁盘上限。

列表规则：

- 所有列表必须分页。
- 不能一次性全拉。
- 必须支持筛选和排序。

审计规则：

- 不记录密码。
- 不记录 token。
- 不记录密钥。
- 只记录脱敏后的操作摘要。

## 22. 最后需要你确认的小点

这两个不影响工程开工，但上线前要定：

1. 正式域名叫什么。
2. 员工看到的受限现金流页面叫“收款状态”还是“订单收款”。
